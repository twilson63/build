Migration Plan

Gleam Architecture Migration

Build goes from React monolith to Gleam actors + Lustre components — with 1:1 feature, look, and feel parity.

Same URL. Same experience. Architecture upgrade, not rewrite.

1 Migration Goal

One-to-one parity. Open the React version and the Gleam version side by side — you should not be able to tell the difference. Same CSS, same layout, same interactions, same timing. This is an architecture upgrade, not a rewrite.

The Gleam migration is not about changing what Build does. It's about changing how Build is built. Every feature, every pixel, every interaction must survive the migration intact.

  • Visual parity — Screenshot comparison passes. Same CSS class names, same DOM structure, same layout.
  • Feature parity — Every interaction works identically: agent calls, project CRUD, file editing, terminal, preview inspector, auto-save, settings.
  • Performance parity — No regressions. xterm streaming, CodeMirror responsiveness, WebContainer boot time.
  • URL parity — Same URL, same deployment target. Swap the JS bundle, nothing else changes.

2 Architecture Overview

Domain Actors

Six self-contained modules, each owning state, messages, and effects. Pure update() functions. No side effects in state transitions.

4 CSS Strategy

styles.css is copied verbatim. No CSS-in-JS, no CSS modules, no shadow DOM styles. Lustre renders real DOM with real class names — the existing CSS works as-is.

Why Light DOM

Lustre's register() creates web components with shadow DOM by default. For Build, we use light DOM components instead. Reason:

  • Shadow DOM isolates styles. The existing styles.css targets .app, .sidebar, .editor, etc. These selectors would not pierce shadow DOM boundaries.
  • Light DOM keeps CSS working. All existing class-based selectors continue to match elements inside <build-*> custom elements.
  • No style duplication. We don't need to copy/translate CSS into shadow DOM stylesheets or ::part() selectors.

How Light DOM in Lustre

Lustre v5.6 supports light DOM components via component.default_slot() and component.named_slot() without attaching a shadow root. The custom element registers with the browser but renders its view into its own children — where global CSS applies.

Before: React JSX
<div className="app">
  <div className="sidebar">
    <div className="editor">
      {selectedFile && (
        <CodeEditor
          value={{selectedFile.content}}
          onChange={{handleEdit}}
        />
      )}</div>
    </div>
  </div>
After: Lustre View
fn view(model) {
  let selected = get_selected_file(model)
  html.div([attr.class("app")], [
    html.div([attr.class("sidebar")], [
      html.div([attr.class("editor")], [
        case selected {
          Some(file) -> html.custom(
            "build-editor",
            [attr.property("value", file.content)],
            [],
          )
          None -> html.text("")
        },
      ]),
    ]),
  ])
}

Same class names, same DOM structure. CSS selectors still match. No ::part(), no :host(), no shadow DOM style piercings needed.

5 CodeMirror & xterm Interop

The two most complex FFI bindings. Both are vanilla JS libraries that need imperative lifecycle management — exactly what Lustre custom elements are designed for.

CodeMirror 6
xterm.js

build-editor Custom Element

CodeMirror 6 is a vanilla JS library — no React dependency. The current CodeEditor.tsx is just 31 lines wrapping @uiw/react-codemirror. In Gleam, we use vanilla CodeMirror 6 directly inside a Lustre custom element.

Attribute/Event Interface

<build-editor> Attributes (input): value String ← file content to display language String ← "javascript" | "html" | "css" | "json" readonly Bool ← disable editing theme String ← "dark" (matches existing CSS) Events (output): change String → emitted on content change, detail = new value cursor-change Int → emitted on cursor position change Lifecycle: connectedCallback → create EditorView with extensions disconnectedCallback → destroy EditorView on_attribute_change("value") → dispatch EditorView.dispatch(replaceAll) on_attribute_change("language") → reload language extension

FFI Binding Pattern

TypeScript (before)
// CodeEditor.tsx — React wrapper

  from '@uiw/react-codemirror'

export function CodeEditor({
  value, onChange, language
}) {
  return (
    <CodeMirror
      value={value}
      onChange={onChange}
      theme="dark"
      extensions={[lang(language)]}

Gleam FFI (after)
// ffi/codemirror.js
export function createEditor(
  container, value, language, theme,
  onChange
) {
  const view = new EditorView({
    state: EditorState.create({
      doc: value,
      extensions: [
        getLang(language),
        keymap(defaultKeymap),
        EditorView.updateListener.of(
          update => {
            if (update.docChanged)
              onChange(update.state.doc.toString())
          }
        ),
      ],
    }),
    parent: container,
  })
  return view
}

export function destroyEditor(view) {
  view.destroy()
}

export function setContent(view, content) {
  view.dispatch({
    changes: {
      from: 0,
      to: view.state.doc.length,
      insert: content,
    },
  })
}
Lustre custom element registration
pub fn register_build_editor() { lustre.register( "build-editor", init_editor, update_editor, view_editor, ) |> component.on_attribute_change("value", decode_string, UpdateValue) |> component.on_attribute_change("language", decode_string, UpdateLanguage) |> component.on_property_change("readonly", decode_bool, SetReadonly) |> lustre.register_ok() } fn init_editor() { // Create CodeMirror instance via FFI let editor = codemirror_ffi.create_editor("", "javascript", "dark") #(EditorModel(editor, "", "javascript"), []) } fn update_editor(model, msg) { case msg { UpdateValue(new_value) -> { // Only update if value actually changed (avoid cursor reset) let _ = codemirror_ffi.set_content(model.editor, new_value) #(..model, content: new_value, []) } UpdateLanguage(lang) -> { let _ = codemirror_ffi.reload_language(model.editor, lang) #(..model, language: lang, []) } SetReadonly(ro) -> { let _ = codemirror_ffi.set_readonly(model.editor, ro) #(model, []) } ContentChanged(new_content) -> { // Emit custom event back to parent #(..model, content: new_content, [ effect.from(emit_change(new_content)) ]) } } } fn emit_change(content) { fn() { let _ = event.emit("change", content) Nil } }

build-terminal Custom Element

xterm.js is imperative by nature — you write to it, you resize it, you attach a FitAddon. Perfect fit for a Lustre custom element with FFI.

Attribute/Event Interface

<build-terminal> Attributes (input): logs List(String) ← log lines to write visible Bool ← show/hide terminal panel Methods (imperative, via FFI): write String → Nil ← write a line to the terminal clear Nil → Nil ← clear terminal buffer fit Nil → Nil ← trigger FitAddon.fit() Lifecycle: connectedCallback → create Terminal + FitAddon + attach to container disconnectedCallback → dispose Terminal on_attribute_change("visible") → show/hide container div effect.before_paint → fit() on resize

FFI Binding Pattern

TypeScript (before)
// TerminalPanel.tsx — React
const termRef = useRef(null)
const fitRef  = useRef(new FitAddon())

useEffect(() => {
  if (!termRef.current) return
  const term = new Terminal()
  term.loadAddon(fitRef.current)
  term.open(termRef.current)
  fitRef.current.fit()
  return () => term.dispose()
}, [])

useEffect(() => {
  const term = termRef.current
  logs.forEach(l => term.writeln(l))
}, [logs])
Gleam FFI (after)
// ffi/xterm.js
export function createTerminal(container) {
  const term = new Terminal({
    theme: { background: '#1a1a2e' },
    fontSize: 13,
  })
  const fit = new FitAddon()
  term.loadAddon(fit)
  term.open(container)
  fit.fit()
  return { term, fit }
}

export function writeLine(terminal, line) {
  terminal.term.writeln(line)
}

export function fitTerminal(terminal) {
  terminal.fit.fit()
}

export function disposeTerminal(terminal) {
  terminal.term.dispose()
}

export function clearTerminal(terminal) {
  terminal.term.clear()
}

Streaming Performance

WebContainer writes logs character-by-character through onStdout. The FFI binding buffers writes and flushes on effect.before_paint to avoid per-character state updates:

fn handle_log(model, line) { // Don't update state per line — buffer and flush let buffered = [string.append(model.pending_log, line), ..model.log_buffer] case list.length(buffered) > 20 { True -> #(..model, log_buffer: [], [ effect.from(flush_logs(buffered)) ]) False -> #(..model, log_buffer: buffered, []) } } fn flush_logs(lines) { fn() { let _ = xterm_ffi.write_lines(terminal, lines) Nil } }

6 The Plugin System

Inspired by CodeMirror's compartment system, each Build feature is a self-contained plug-in. New features are added as <build-*> custom elements without modifying core.

Open-closed principle. Add <build-diff-view>, <build-deploy>, or <build-git-status> without touching a single line of app.gleam. They register as custom elements, communicate via the inter-component bus, and compose in the shell.

Plugin Architecture

<build-app> ← shell: reads AppState, dispatches Msg, renders children │ ├── <build-editor> ← core plugin ├── <build-terminal> ← core plugin ├── <build-preview> ← core plugin ├── <build-agent-chat> ← core plugin ├── <build-project-nav> ← core plugin ├── <build-element-picker> ← core plugin │ ├── <build-diff-view> ← future plugin ├── <build-deploy> ← future plugin ├── <build-git-status> ← future plugin └── <build-test-runner> ← future plugin Communication: effect.provide("build:state", state_json) ← app publishes state effect.subscribe("build:state", decoder) ← plugins subscribe event.emit("build:msg", msg_json) ← plugins emit messages

Core Plugins

<build-editor>
CodeMirror 6 vanilla editor. Attributes: value, language. Events: change.
<build-terminal>
xterm.js terminal. Attributes: visible. Methods: write, clear, fit.
<build-preview>
iframe management. Attributes: src. Events: element-selected. Inspector postMessage bridge.

<build-project-nav>

Ordered phases with verification gates. Each phase ends with a concrete pass/fail check before moving on.

Phase 1 Port Pure Modules

Move the easy stuff first — pure functions with no side effects and no React dependency.

  • templates.ts → templates.gleam — pure data, no FFI
  • preview-inspector.ts → preview_inspector.gleam — pure types/functions
  • design-guidance.ts → design_guidance.gleam — pure function
  • editor.ts → editor.gleam — pure function

These are the safest ports. No FFI, no DOM, no side effects. Validates that Gleam's JavaScript target works for the data types we need.

All 4 modules ported. Gleam unit tests pass. TypeScript tests still pass (we haven't deleted the originals yet).

Phase 2 Build Actor System — Simple Actors

Start with the stateless actors: Settings, Chat, Preview. These have minimal effects and no cross-actor dependencies.

  • actors/chat.gleam — ChatState, ChatMsg (no effects — Effect is Nil)

3 actor modules with 100% exhaustiveness checks. All actor unit tests pass. Gleam compiler catches missing cases.

Phase 3 Build Complex Actors

async function submit() {

  setAgentStartedAt(Date.now())

Gleam — agent.gleam + app.gleam (after)
// agent.gleam — pure update, 15 lines
fn update(state, msg) {
  case msg {
    AgentRequestStarted(at) -> #(
      ..state, lifecycle: Running(at),
      [],
      [StartElapsedTimer, CallAgent(..)],
    )
    AgentRequestSucceeded(reply, patches) -> #(
      ..state, lifecycle: Idle,
      [StopElapsedTimer,
       InstallIfNeeded(patches)],
    )
    AgentRequestFailed(msg) -> #(
      ..state, lifecycle: Idle,
      [StopElapsedTimer],
    )
  }
}

// app.gleam — composes cross-actor effects
fn app_update(state, msg) {
  case msg {
    SubmitPrompt -> {
      let #(chat, chat_fx) = chat.update(
        state.chat, UserSentMessage(prompt)
      )
      let #(agent, agent_fx) = agent.update(
        state.agent, AgentRequestStarted(now())
      )
      #(..state, chat:, agent:,
        [agent_fx, chat_fx])
    }
  }
}

9 Risk Assessment

MEDIUM

FFI Complexity

Five JS modules stay as FFI bindings (agent, projects, webcontainer, zip, CodeMirror). Each needs carefully typed Gleam externals + JS implementations.

Start with FFI in Phase 1 (pure modules) where risk is zero. Build CodeMirror/xterm FFI in Phase 5 when components are isolated. FFI surface area is small — each binding is ~50-80 lines of JS.
MEDIUM

Lustre Maturity

Lustre is v5.6 — active development, not yet 1.0. Custom elements API may change. Documentation gaps exist.

Lustre's core architecture (Elm Architecture) is battle-tested — the pattern is 10+ years old. Custom element registration is a thin layer. If Lustre breaks, the actor system and state management are independent and portable. Worst case: re-render with vanilla custom elements + actor subscriptions.
LOW

CSS Parity

Shadow DOM could break CSS selectors. Existing CSS targets flat class names.

Use light DOM for all Build components. Same class names, same DOM nesting, same CSS. Shadow DOM is opt-in — we opt out. Verified in Phase 7 (visual parity gate).
MEDIUM

xterm Streaming Performance

WebContainer writes stdout character-by-character. Per-character state updates would kill performance.

The xterm custom element writes directly to the Terminal instance via FFI — bypassing Lustre's state for streaming writes. Log state is maintained for replay, but the terminal display is imperative. Buffer + flush on requestAnimationFrame via effect.before_paint.
LOW

CodeMirror FFI

CodeMirror 6 has a complex extension system. Mapping it through FFI could be fragile.

We only use basic extensions: language support, keymap, dark theme. The FFI binding exposes create, setContent, destroy, reloadLanguage — a narrow surface. Extensions are configured in JS, not exposed to Gleam.
MEDIUM

WebContainer API FFI

WebContainer API is async, event-driven, and browser-only. Boot sequence involves multiple async steps.

The WebContainer FFI module mirrors the existing webcontainer.ts — same API calls, same event handlers. The Gleam side only sees typed Effects. The JS side handles all async complexity. This is identical to how the React version works, just moved from useEffect to effect runtime.
MEDIUM

Dev Tooling

Vite has hot reload, TypeScript checking, and mature debugging. lustre_dev_tools is newer and less feature-complete.

Use Vite as the bundler (Gleam compiles to JS, Vite bundles it). lustre_dev_tools for dev server during development. Production builds go through Gleam compile → JS output → Vite bundle. We keep Vite — it's not a Vite vs lustre_dev_tools choice.

Production Build

10 Decision: Why Gleam

Not "why functional programming" or "why type safety" — those are general arguments. Why Gleam specifically for Build.

1. The actor model matches Build's domain

Build has 6 distinct state domains (Settings, Project, Chat, Agent, Preview, WebContainer) that currently share a 545-line component with 19 useState hooks. Gleam's actor model — where each domain is a self-contained module with its own State/Msg/Effect — maps 1:1 to Build's actual structure. This isn't forcing a pattern; it's revealing the pattern that was always there.

2. Exhaustiveness checking catches real bugs

The current code has implicit state machines (busy + agentStartedAt, projectReady + hydrated). When these get out of sync, bugs happen. Gleam's never-typed default branches make missing cases a compile error. This isn't theoretical — the current code's busy flag can race between WebContainer boot and agent calls. The state machine approach makes this impossible.

3. Lustre components are Build's plugin architecture

Build wants to add diff views, deploy buttons, git status, test runners. With React, each new feature means editing main.tsx. With Lustre custom elements, each feature is <build-*> — register it, drop it in the shell, done. The component bus lets plugins subscribe to state and emit messages without touching core. This is the extensibility model Build needs.

4. Zero runtime dependency addition

Gleam compiles to plain JavaScript. Lustre is ~10KB. We're removing react + react-dom (~40KB) and @uiw/react-codemirror (~15KB). The net effect is a smaller bundle with a smaller dependency tree. No Zustand, no XState, no external state manager — the actor pattern is the state manager.

5. Pure functions are testable functions

Every update() is a pure function: update(state, msg) → (state, effects). No React test renderer. No hook mocking. No async setup. Just call the function, check the result. This is how the existing 42 tests work on pure modules — we're extending the same pattern to state management.

The short version: Gleam doesn't add complexity — it removes it. The 545-line monolith becomes 15 focused modules. The 19 useState hooks become 6 typed actors. The copy-pasted agent code becomes one Effect. And the whole thing gets a plugin system for free.
